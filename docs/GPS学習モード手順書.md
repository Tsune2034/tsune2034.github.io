# GPS学習モード 手順書

**対象:** KAIROX ドライバー・管理者
**作成日:** 2026-03-25
**バージョン:** v1.0

---

## 概要

GPS学習モードとは、配送中のドライバーのGPS軌跡を記録・分析し、ルートごとの実走行時間を蓄積することで、ETA（到着予定時刻）の精度を自動改善する仕組みです。

```
走るたびにデータが蓄積 → ルートごとの補正係数が計算される → ETAが実態に即した値になる
```

### 学習の流れ

```
配送開始
  └─ GPS 30秒ごとに全点記録（gps_tracks テーブル）
         ↓
配送完了（「配達完了」ボタン）
  └─ 実走行時間を自動計算
  └─ ルート統計（route_stats）に加重平均で蓄積
         ↓
同ルート 3件以上蓄積
  └─ ETA計算に補正係数が自動適用（source: "learned" と表示）
```

## 1. 事前準備

### 1-1. テスト予約の作成

1. ブラウザで `/narita` を開く
2. 通常通り予約を入力する
   - **pickup:** 配送出発地点（例: 成田空港第1ターミナル）
   - **destination:** 配送先（例: 千葉市内のホテル）
   - **pickup_date:** 当日の日時
3. 予約完了 → **予約番号（KRX-XXXXXX）をメモしておく**

> テスト用なので支払いは Stripe テストカードで可。金額が発生しない場合は管理画面から手動で予約を作成してもよい。

### 1-2. ドライバー画面の準備

1. スマートフォンで `/driver` を開く
2. PIN 入力（`NEXT_PUBLIC_DRIVER_PIN` の値）
3. **該当予約を選択**
4. ブラウザの GPS 使用許可を「許可」にする
   - iOS Safari: 設定 > Safari > 位置情報 → 許可
   - Android Chrome: アドレスバー左の鍵マーク → 位置情報 → 許可

## 2. 学習テストの実施手順

### Step 1: 出発前に「向かっています」を押す

ドライバー画面で対象予約の **「向かっています」** ボタンをタップ。

- この時点から GPS 記録が開始される
- 30秒ごとに位置情報がサーバーへ送信・保存される

### Step 2: そのまま走行する

スマートフォンをダッシュボードに置いたまま走行。特別な操作は不要。

**注意事項:**
- 画面がオフになってもバックグラウンドで送信される（ブラウザの設定による）
- トンネルや地下など GPS が途切れる区間は自動でスキップされる
- アプリを閉じると送信が止まる。画面はオンのまま維持推奨

### Step 3: 到着後に「配達完了」を押す

目的地（ホテルなど）に到着し荷物を渡したら **「配達完了」** ボタンをタップ。

- このボタンを押した瞬間に以下が実行される:
  1. GPS 記録の終了
  2. 出発〜到着の実走行時間を計算
  3. `route_stats` テーブルに保存（ルート + 時間帯 + 実走行時間）

## 3. 学習データの確認方法

### 管理者APIで確認（Railway バックエンド）

```
GET /route-stats
```

レスポンス例:
```json
[
  {
    "pickup_grid": "35.8,140.4",
    "dest_grid": "35.6,140.1",
    "hour_of_day": 14,
    "avg_actual_min": 78.3,
    "google_est_min": 65.0,
    "correction_factor": 1.205,
    "sample_count": 3,
    "updated_at": "2026-03-25T14:32:00Z"
  }
]
```

### フィールドの見方

| フィールド | 説明 |
|---|---|
| `pickup_grid` | 出発地の0.1度グリッド座標（緯度,経度） |
| `dest_grid` | 目的地の0.1度グリッド座標 |
| `hour_of_day` | 出発時刻（JST, 0〜23） |
| `avg_actual_min` | このルート・時間帯の実走行時間平均（分） |
| `google_est_min` | Google API の推定値（分） |
| `correction_factor` | 補正係数（avg_actual / google_est）|
| `sample_count` | サンプル件数 |

### 補正係数の意味

| correction_factor | 意味 |
|---|---|
| 1.0 | Google の推定通り |
| 1.2 | Google より20%遅い（渋滞・駐停車等） |
| 0.9 | Google より10%速い（深夜・高速利用等） |

---

## 4. ETAへの反映タイミング

| サンプル数 | 動作 |
|---|---|
| 0〜2件 | 補正なし（Google API の値をそのまま表示） |
| 3件以上 | 補正係数が自動適用（TrackingView に `learned` と表示） |
| 20件以上（将来実装） | 時間帯別（朝/昼/夜/深夜）に細分化された補正 |

ETA が学習補正済みの場合、TrackingView の到着予測に内部的に `source: "learned"` が付与される。

## 5. テスト走行の推奨プラン（千葉テスト）

成田空港 → 千葉市内ルートのデータを蓄積するプラン:

| 回 | 出発時間帯 | 目的 |
|---|---|---|
| 1回目 | 昼（12〜14時） | 基準データ取得 |
| 2回目 | 夕方（17〜19時） | 混雑時間帯の実態把握 |
| 3回目 | 朝（7〜9時） | **3件揃い、補正係数が自動適用開始** |
| 5回目〜 | 任意 | 精度向上（加重平均が安定） |

3回走れば本番ETA に補正が効き始める。

## 6. トラブルシューティング

### GPS が記録されない

- ブラウザの位置情報許可を確認
- HTTP（非HTTPS）では GPS が動作しない → 本番URL（https://...）を使用
- iOSの場合: 設定 > プライバシー > 位置情報サービス > Safari → 「このAppの使用中」

### 「配達完了」を押してもデータが保存されない

- GPS点が3点未満の場合は保存されない（5分未満の走行は除外）
- バックエンドログで確認: Railway ダッシュボード > Logs > `[RouteLearn]` の行を確認

### 補正係数が1.0のまま変わらない

- `sample_count` が3未満 → あと何回か走る必要がある
- `GET /route-stats` でデータを確認し、`pickup_grid` と `dest_grid` が正しい値か確認

## 7. 技術仕様（エンジニア向け）

### 関連テーブル

```sql
-- GPS履歴（配送1件につき複数行）
gps_tracks:
  id, booking_id, lat, lng, driver_status, recorded_at

-- ルート統計（ルート+時間帯ごとに1行）
route_stats:
  id, pickup_grid, dest_grid, hour_of_day,
  actual_min, avg_actual_min, google_est_min,
  correction_factor, sample_count, updated_at
```

### 関連エンドポイント

```
PUT /bookings/{id}/driver-location   GPS点を保存 + 完了時に自動解析
GET /route-stats                     学習データ一覧
GET /route-stats/correction          ETA補正係数取得（フロントエンドが呼ぶ）
```

### グリッド精度

座標を 0.1度（約11km）単位に丸めてルートを識別。
成田空港は `35.8,140.4`、千葉市内は `35.6,140.1` のように識別される。
