# GPS学習モード 手順書

**対象:** KAIROX ドライバー・管理者
**作成日:** 2026-03-25
**バージョン:** v1.1（2026-03-25 ルート種別学習追加）

---

## 概要

GPS学習モードとは、配送中のドライバーのGPS軌跡を記録・分析し、ルートごとの実走行時間を蓄積することで、ETA（到着予定時刻）の精度を自動改善する仕組みです。

**v1.1追加:** 高速道路と一般道を**別々に学習**できるルート種別機能を追加。往復収益の最適化（「高速で急いで戻るべきか、一般道でコストを抑えるべきか」）の判断データとして活用できます。

```
走るたびにデータが蓄積 → ルート×時間帯×道路種別ごとの補正係数が計算される
                       → ETAが実態に即した値になる
```

### 学習の流れ

```
出発前にルート種別を選択（🛣️ 高速 / 🏘️ 一般道）
  │
  ▼
配送開始
  └─ GPS 30秒ごとに全点記録（route_type付きで gps_tracks テーブルへ）
         ↓
配送完了（「配達完了」ボタン）
  └─ 実走行時間を自動計算
  └─ ルート統計（route_stats）に route_type 別で加重平均を蓄積
         ↓
同ルート × 同種別 3件以上蓄積
  └─ ETA計算に補正係数が自動適用（source: "learned" と表示）
```

## 1. 事前準備

### 1-1. テスト予約の作成

1. ブラウザで `/narita` を開く
2. 通常通り予約を入力する
   - **pickup:** 配送出発地点（例: 成田空港第1ターミナル）
   - **destination:** 配送先（例: 千葉市内のホテル）
   - **pickup_date:** 当日の日時
3. 予約完了 → **予約番号（KRX-XXXXXX）をメモしておく**

> テスト用なので支払いは Stripe テストカードで可。金額が発生しない場合は管理画面から手動で予約を作成してもよい。

### 1-2. ドライバー画面の準備

1. スマートフォンで `/driver` を開く
2. PIN 入力（`NEXT_PUBLIC_DRIVER_PIN` の値）
3. **該当予約を選択**
4. ブラウザの GPS 使用許可を「許可」にする
   - iOS Safari: 設定 > Safari > 位置情報 → 許可
   - Android Chrome: アドレスバー左の鍵マーク → 位置情報 → 許可

## 2. 学習テストの実施手順

### Step 1: ルート種別を選択する（出発前・必須）

ドライバー画面の配送カードに **ルート種別トグルボタン** が表示される。

```
┌──────────────────────────────┐
│  KRX-XXXXXX   🚐 向かっています  │
│                               │
│  🛣️  高速道路      [タップで切替] │  ← ここをタップ
│     学習: 高速ルート             │
│                               │
│  ● GPS追跡を開始                │
└──────────────────────────────┘
```

- **🛣️ 高速道路（紫）:** 高速道路経由で走る場合
- **🏘️ 一般道（緑）:** 一般道経由で走る場合

> 走行中に切り替えることも可能。最後の設定値がそのトリップの route_type として記録される。

### Step 2: GPS をONにして出発する

ルート種別を設定したら **「GPS追跡を開始」** をタップしてからステータスを「向かっています」に更新。

- GPS ON にした時点から30秒ごとに位置情報がサーバーへ保存される
- ルート種別も各GPS点と一緒に記録される

**走行中の注意事項:**
- 画面はオンのまま維持推奨（バックグラウンドでの動作はブラウザ設定による）
- トンネル・地下などGPSが途切れる区間は自動スキップされる
- アプリを閉じると送信が止まる

### Step 3: 到着後に「配達完了」を押す

目的地に到着し荷物を渡したら **「配達完了」** ボタンをタップ。

- このボタンを押した瞬間に以下が実行される:
  1. GPS 記録の終了
  2. 出発〜到着の実走行時間を計算
  3. `route_stats` テーブルに **route_type 別** で保存（高速と一般道は別レコード）
  4. Railwayログに `[RouteLearn]` として記録される

## 3. 学習データの確認方法

### 管理者APIで確認（Railway バックエンド）

```
GET /route-stats
```

レスポンス例:
```json
[
  {
    "pickup_grid": "35.8,140.4",
    "dest_grid": "35.6,140.1",
    "route_type": "highway",
    "hour_of_day": 14,
    "avg_actual_min": 52.1,
    "google_est_min": 48.0,
    "correction_factor": 1.086,
    "sample_count": 2,
    "updated_at": "2026-03-25T14:32:00Z"
  },
  {
    "pickup_grid": "35.8,140.4",
    "dest_grid": "35.6,140.1",
    "route_type": "local",
    "hour_of_day": 14,
    "avg_actual_min": 78.3,
    "google_est_min": 65.0,
    "correction_factor": 1.205,
    "sample_count": 3,
    "updated_at": "2026-03-25T10:15:00Z"
  }
]
```

同じ出発地・目的地・時間帯でも **highway と local は別レコード** として蓄積される。

### フィールドの見方

| フィールド | 説明 |
|---|---|
| `pickup_grid` | 出発地の0.1度グリッド座標（緯度,経度） |
| `dest_grid` | 目的地の0.1度グリッド座標 |
| `route_type` | **highway**（高速道路経由）/ **local**（一般道経由） |
| `hour_of_day` | 出発時刻（JST, 0〜23） |
| `avg_actual_min` | このルート・時間帯・道路種別の実走行時間平均（分） |
| `google_est_min` | Google API の推定値（分） |
| `correction_factor` | 補正係数（avg_actual / google_est）|
| `sample_count` | サンプル件数 |

### 補正係数の意味

| correction_factor | 意味 |
|---|---|
| 1.0 | Google の推定通り |
| 1.2 | Google より20%遅い（渋滞・駐停車等） |
| 0.9 | Google より10%速い |

### 高速 vs 一般道の比較活用例

同じ成田→千葉 昼14時帯で学習データが揃った場合:

| route_type | avg_actual_min | 高速代 | 実質コスト |
|---|---|---|---|
| highway | 52分 | ¥1,500 | ¥1,500 |
| local | 78分 | ¥0 | ¥0 |

→ 「次の便まで残り70分」なら高速推奨、「次の便まで残り90分」なら一般道でコスト節約、という判断が自動化できる（将来実装）。

## 4. ETAへの反映タイミング

| サンプル数 | 動作 |
|---|---|
| 0〜2件 | 補正なし（Google API の値をそのまま表示） |
| 3件以上 | 補正係数が自動適用（TrackingView に `learned` と表示） |
| 20件以上（将来実装） | 時間帯別（朝/昼/夜/深夜）に細分化された補正 |

ETA が学習補正済みの場合、TrackingView の到着予測に内部的に `source: "learned"` が付与される。

## 5. テスト走行の推奨プラン（千葉テスト）

成田空港 → 千葉市内ルートのデータを蓄積するプラン。
高速と一般道を**それぞれ3件ずつ**集めることを目標にする。

### 優先順位付き走行計画

| 回 | ルート種別 | 出発時間帯 | 目的 |
|---|---|---|---|
| 1回目 | 🛣️ 高速 | 昼（12〜14時） | 高速の基準データ取得 |
| 2回目 | 🏘️ 一般道 | 昼（12〜14時） | 同条件で比較データ取得 |
| 3回目 | 🛣️ 高速 | 夕方（17〜19時） | 混雑時間帯の高速 |
| 4回目 | 🏘️ 一般道 | 夕方（17〜19時） | 混雑時間帯の一般道 |
| 5回目 | 🛣️ 高速 | 朝（7〜9時） | **高速3件目 → 補正適用開始** |
| 6回目 | 🏘️ 一般道 | 朝（7〜9時） | **一般道3件目 → 補正適用開始** |
| 7回目〜 | どちらか | 任意 | 精度向上（加重平均が安定化） |

### 最速で効果を出す場合

時間がない場合は同じルート種別を3回連続で走ればOK。

```
例: 高速で3回走る
  → 高速ルートのETA補正が3回目完了後から適用される
  → 一般道データはまだないので、一般道走行時はGoogle推定値のまま
```

### 往復収益判断に使えるデータが揃う条件

高速・一般道それぞれ3件ずつ同じ時間帯に集まると、
「この時間帯に高速を使うべきか」の計算が自動化できる段階になる。

## 6. トラブルシューティング

### GPS が記録されない

- ブラウザの位置情報許可を確認
- HTTP（非HTTPS）では GPS が動作しない → 本番URL（https://...）を使用
- iOSの場合: 設定 > プライバシー > 位置情報サービス > Safari → 「このAppの使用中」

### 「配達完了」を押してもデータが保存されない

- GPS点が3点未満の場合は保存されない（5分未満の走行は除外）
- バックエンドログで確認: Railway ダッシュボード > Logs > `[RouteLearn]` の行を確認

### 補正係数が1.0のまま変わらない

- `sample_count` が3未満 → あと何回か走る必要がある
- `GET /route-stats` でデータを確認し、`pickup_grid`・`dest_grid`・`route_type` が正しい値か確認
- **高速と一般道は別カウント。** 高速3件・一般道2件の場合、高速のみ補正が適用され一般道はまだ適用されない

### ルート種別が意図せず変わってしまった

- 走行中にトグルを誤タップすると route_type が変わる
- 最後にタップした状態が記録される
- 誤った場合はそのトリップのデータを無視して再走行する（`GET /route-stats` で sample_count が増えていなければ問題なし）

## 7. 技術仕様（エンジニア向け）

### 関連テーブル

```sql
-- GPS履歴（配送1件につき複数行）
gps_tracks:
  id, booking_id, lat, lng, driver_status,
  route_type,   -- "highway" | "local"
  recorded_at

-- ルート統計（ルート + 時間帯 + 道路種別ごとに1行）
route_stats:
  id, pickup_grid, dest_grid,
  route_type,   -- "highway" | "local"（高速と一般道は別レコード）
  hour_of_day,
  actual_min, avg_actual_min, google_est_min,
  correction_factor, sample_count, updated_at
```

### 関連エンドポイント

```
PUT /bookings/{id}/driver-location?route_type=highway
                                     GPS点を保存 + 完了時に自動解析
GET /route-stats                     学習データ一覧（全ルート種別）
GET /route-stats/correction?olat=...&dlat=...&hour=14&route_type=highway
                                     ETA補正係数取得（フロントエンドが呼ぶ）
```

### グリッド精度

座標を 0.1度（約11km）単位に丸めてルートを識別。
成田空港は `35.8,140.4`、千葉市内は `35.6,140.1` のように識別される。

### ルート識別キーの構成

`pickup_grid` + `dest_grid` + `route_type` + `hour_of_day` の4つが一致したレコードを同一ルートとして加重平均する。
