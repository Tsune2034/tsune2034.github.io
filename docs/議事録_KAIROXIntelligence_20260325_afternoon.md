## 2. /traffic マップ データ収集戦略

### 2-1. 現在のデータ構造（Engineer報告）

```
GpsTrackPoint テーブル（PostgreSQL on Railway）
  booking_id, lat, lng, driver_status, route_type, recorded_at
  → 30秒間隔で記録
  → route_type: "highway" | "local"（Tsuneが手動切替）

CongestionSegment テーブル
  grid(0.05°), lat, lng, route_type, hour_of_day, day_of_week
  avg_speed_kmh, congestion_level, sample_count
  → GPS点から自動計算・蓄積

RouteStats テーブル
  pickup_grid, dest_grid, route_type, hour_of_day
  avg_actual_min, correction_factor, sample_count
  → ETA精度向上に使用
```

### 2-2. テストデータの考え方（Strategy + Engineer）

**現状（データ0件）の対応:**
- Google TrafficLayer がベースライン表示（関東全域カバー）

- Day 1からマップとして機能している（Googleが穴を埋めている）

**データ蓄積ロードマップ:**

| フェーズ | 件数 | 期間（目安） | できること |

| Phase 1 | 0〜20件 | 〜1ヶ月 | Googleレイヤーのみ表示。蓄積中 |

## 5. Devil's Advocate — リスク論証

**Tsune、以下のリスクを認識しておく:**

1. **Googleマップへの依存リスク** — Google Maps API の利用規約変更・価格改定で一夜にしてコスト構造が壊れる可能性。独自描画への移行オプション（Mapbox等）を将来検討すべき

2. **データ蓄積の速度リスク** — Tsuneが1人で走るだけでは蓄積が遅い。ドライバーが増えれば自動解決するが、その前にユーザーが来てもデータが薄い状態が続く

3. **AdSense審査の不確実性** — 近年審査が厳しい。落ちた場合の代替（A8.net等のアフィリエイト）も準備すべき

4. **コンテンツの陳腐化** — リアルタイム渋滞情報はデータが古くなると価値がゼロになる。60秒自動更新は実装済みだが、バックエンドのデータが古ければ表示も古い
