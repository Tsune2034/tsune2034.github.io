# 議事録 — KAIROX サービス拡張戦略会議

**日時:** 2026-03-23
**テーマ:** 「成田到着→即観光」モデルへの転換・ホテル検索全網羅・空港間輸送・ドライバー空港情報の設計
**提起者:** Tsune
**参加者:** planner / research / engineer / devils advocate / optimizer / cto / product manager / marketing / finance / strategy

---

## 会議の背景（CEO発言）

> 海外から成田に到着した旅行者は、本来ならホテルへ行ってチェックインしてから観光になる。しかし KAIROXでキャリーケースをホテルへ先に届ければ、成田到着から即観光が可能になる。半日スケジュールが前倒しになる。円安で旅行予算が使いやすい今こそこの価値が刺さる。
>
> やりたいこと：
> 1. 成田から千葉・都内全ホテルを検索・選択できる仕様
> 2. ドライバー側への空港情報提供（フライト・到着状況）
> 3. 羽田↔成田の空港間輸送
> 4. 成田・羽田から北海道への接続も取れる構造

### research（市場調査）

CEOの仮説を数字で裏付ける。

**「半日の価値」の定量化**
- 成田空港から都内ホテルまでの移動時間: 平均 90〜120分（電車 or バス）
- チェックインまでの待機: 15〜30分
- 荷物を持った状態では観光への行動移行コストが高い
- KAIROXを使えば到着直後から **平均3〜4時間の観光時間を創出** できる

**円安効果（2026年3月時点）**
- USD/JPY: 約148〜152円
- 米国人旅行者の実質購買力: 2020年比 +25〜30%
- 「もう1日分観光できる」という心理的余裕が生まれる
- 特に初訪日・短期滞在（3〜5泊）層に刺さるバリュープロポジション

**空港間輸送の市場**
- 成田↔羽田のアクセスは悩みのトップ3に入る旅行者課題
- 現状: バス（1,300円・70分）、電車（3,000〜4,000円・90〜110分）
- 荷物持ちでの乗り換えは特に苦痛
- 成田→羽田の荷物先送りなら旅行者は **身軽に電車移動** できる

### planner（計画設計）

サービス全体を「荷物の旅」として再設計する。旅行者の旅程に沿って KAIROX が使える場面を整理する。

```
【旅程ジャーニーマップ】

Day 1 成田到着
  ↓ KAIROX: 成田 → 都内ホテル（本日の中心施策）
  ↓ 旅行者は手ぶらで成田周辺 or 成田エクスプレスで即観光へ
  ↓ ホテルには荷物が先着

Day 2〜4 都内観光
  ↓ KAIROX: 都内ホテル → 次のホテル（泊まり移動の荷物先送り）

Day N 移動日
  ↓ KAIROX: 都内ホテル → 羽田T1/T2/T3（帰国前荷物先送り）
  or
  ↓ KAIROX: 羽田 → 成田（乗り継ぎ荷物）
  or
  ↓ KAIROX: 成田/羽田 → 新千歳（北海道接続）

最終日 帰国
  ↓ KAIROX: ホテル → 成田 or 羽田
```

**必要な機能拡張:**
1. ホテル検索（全国 → まず千葉・東京23区・横浜）
2. 空港→ホテルのゾーン料金設定
3. 空港間輸送ルート（成田↔羽田）
4. 空港→空港の料金体系（固定 or 距離ベース）

### engineer（技術設計）

3つの技術的実装課題に分けて説明する。

#### 課題1: ホテル全網羅検索

現在の DESTINATIONS 配列はエリア単位（銀座・新宿など10エリア）。これを**ホテル単位**に拡張するには2つのアプローチがある。

**A. 静的ホテルリスト（短期実装）**
```typescript
const HOTELS: Hotel[] = [
  { id: "park-hyatt-shinjuku", name: "パークハイアット東京",
    area: "shinjuku", lat: 35.6889, lng: 139.6917, priceJpy: 6200 },
  { id: "aman-tokyo", name: "アマン東京",
    area: "marunouchi", lat: 35.6840, lng: 139.7644, priceJpy: 5700 },
  // ... 主要ホテル200件
]
```
- 実装コスト: 低
- メンテナンスコスト: 高（ホテル増減への対応）

**B. ホテル検索API連携（Phase 2）**
- Google Places API / Foursquare Places API
- ユーザーが「ホテル名」を入力 → オートコンプリート → 座標取得
- 料金はホテル座標からKAIROX本社（または成田）との直線距離で動的計算

**推奨:** Phase 1 は主要200ホテルの静的リスト + フリーテキスト入力。Phase 2 で Places API 連携。

#### 課題2: ドライバー側への空港情報提供

**フライト情報API**
- AviationStack API（月1万リクエスト無料）または FlightAware API
- `GET /flights?airport=NRT&type=arrivals` → 到着便一覧取得
- ドライバーアプリ（DriverView）に「本日の到着便タイムライン」を表示

```
現在実装済み: GET /api/flights/route.ts（プロキシ）
→ バックエンドに GET /flights エンドポイントを追加するだけで接続可能
```

**ドライバー向け情報パネル（DriverView 追加項目）**
- 次の大型到着便（便名・出発地・到着予定時刻・ターミナル）
- 現在の集荷依頼件数
- 各ターミナルの混雑状況（Phase 2）

#### 課題3: 空港間輸送ルート

**成田↔羽田の座標・料金**
```typescript
const AIRPORT_TRANSFERS = [
  {
    from: "NRT", to: "HND",
    distanceKm: 91, etaMin: 95,
    priceJpy: 8500,  // 専用料金（通常エリア配達より高め）
    note: "Airport-to-Airport Express"
  },
  // 逆方向
  { from: "HND", to: "NRT", ... }
]
```

**北海道接続（成田/羽田 → 新千歳）**
- 北海道便は `chitose` ゾーンの既存インフラを活用
- 接続フロー: 成田到着 → KAIROX で荷物を成田クローク or 羽田預かり所へ → 旅行者は手ぶらで搭乗 → 北海道側で別途配送
- Phase 2 の設計（現在の北海道バックエンドとの連携が必要）

### devils advocate（リスク分析・反対意見）

**反論1: 「即観光」は本当に旅行者が求めているか？**

研究では「半日の価値」と言うが、実際の旅行者行動を観察すると、成田到着後は「まずホテルに行きたい」という強いニーズがある。疲れている・シャワーを浴びたい・ホテルで一息つきたい、という行動パターンは根強い。特にファミリー層や長距離フライト旅行者（欧米）は「即観光」より「ホテル先着」を望む割合が高い可能性がある。ユーザーインタビューなしに「半日前倒し」需要を前提にするのは早計。

**反論2: ホテル全網羅は運営コストの罠**

千葉・東京23区のホテル数は数千件に上る。静的リストの200件では網羅率が低く、「私のホテルが載っていない」という離脱を生む。かといって Places API を使えばコストが跳ね上がる。Google Places の料金は1リクエスト 約0.017ドル。月10万リクエストで約1,700ドル（約25万円）。売上が追いついていない段階では固定コスト増になる。

**反論3: 空港間輸送の競合はすでに強い**

成田↔羽田の荷物輸送は **ヤマト運輸のクロネコ空港間宅急便**、**JALのファストバゲージ**がすでに対応。価格は1個 1,500〜2,500円（KAIROX の8,500円と比べて大幅に安い）。ただし「当日配送」「ドライバーが来る」という点では差別化余地がある。

**反論4: ドライバーのフライト情報利用に法的制限がある可能性**

航空会社のリアルタイム到着情報は航空会社の財産権が絡む場合がある。AviationStack のデータは公開ソースだが、商用利用の規約確認が必要。

### optimizer（改善提案）

devils advocateの指摘を踏まえた改善案：

**改善1: 「即観光」の検証は予約データから**
- 現在の予約で `pickup_date` と実際の到着時刻の差を分析
- フライト番号入力からチェックイン不要層を識別
- 3週間後に仮説検証できる設計にする

**改善2: ホテル検索の段階的拡張**
```
Phase 1: エリア選択（現行） + ホテル名フリーテキスト入力
Phase 1.5: 主要50ホテル（五つ星・人気ホテル優先）の静的リスト
Phase 2: Google Places オートコンプリート（コスト管理のため1日100リクエスト上限設定）
```

**改善3: 空港間輸送のポジショニング**
- ヤマトより高いが「当日・ドア to ドア・英語対応」で差別化
- ターゲットは **ビジネス旅行者・ファーストクラス乗客** → 時間の価値が高い層
- 料金を ¥8,500 → ¥12,000 に引き上げてプレミアムポジション確立

**改善4: ドライバー情報パネルの即時実装**
既存の `/api/flights/route.ts` を使い、AviationStack の無料枠（月1万リクエスト）でまず動かす。商用化後に有料プランへ移行。

### cto（技術責任者視点）

今回の拡張要件をアーキテクチャ観点で整理する。

**実装の依存関係（順序）:**
```
1. ホテル検索（フリーテキスト + 静的リスト） ← 最も価値が高く最も簡単
2. ドライバー空港情報パネル ← /api/flights が既存、AviationStack 接続のみ
3. 空港間輸送料金テーブル ← DESTINATIONS 配列に追記するだけ
4. 北海道接続 ← 別バックエンドとの連携が必要、Phase 2
```

**技術負債の注意点:**
現在の `/narita/page.tsx` はすでに1,200行超。ホテル検索・空港間ルートを同じファイルに追加すると保守が困難になる。ここで**コンポーネント分割**を行うことを強く推奨する。

```
/narita/
  page.tsx             ← メインフロー（thin）
  components/
    StepPickup.tsx
    StepDestination.tsx ← ホテル検索はここ
    StepLuggage.tsx
    StepConfirm.tsx
    StepMatching.tsx
    StepLive.tsx
    DriverInfoPanel.tsx  ← 新規
```

### product manager（プロダクト設計）

今回の要件を整理すると、実はKAIROXの**プロダクト定義が変わる**。

**現在のプロダクト定義:**
「成田空港 → 都内ホテルへの荷物配送サービス」

**拡張後のプロダクト定義:**
「旅行者の旅程全体をカバーする荷物フリー移動プラットフォーム」

これは単なる機能追加ではなく、**サービスの本質的な拡大**。

**新しいユースケース一覧:**

| ユースケース | 区間 | 優先度 |
|------------|------|--------|
| 到着即観光 | 成田 → 都内ホテル | P0（現行） |
| 連泊移動 | ホテルA → ホテルB | P1 |
| 帰国前荷物先送り | ホテル → 成田/羽田 | P1 |
| 空港乗り継ぎ | 成田 → 羽田 / 羽田 → 成田 | P1 |
| 北海道接続 | 成田/羽田 → 新千歳 | P2 |
| ホテル → ホテル（地方） | 東京 → 京都 etc. | P3 |

**プロダクトの名前・ブランドの再定義も検討すべき。「成田ベータ版」から「Japan Luggage Freedom」への格上げ。**

### marketing（市場戦略）

**コアメッセージの転換:**

```
Before: 「荷物を届けます」（機能訴求）
After:  「到着したその瞬間から、日本を楽しめる」（体験訴求）
```

**円安 × 即観光 の刺さるコピー:**
- EN: *"Land at Narita. Start exploring. Your luggage meets you at the hotel."*
- JA: *「着いた瞬間から、旅が始まる。」*
- ZH: *「抵达成田，立刻出发。行李先到酒店。」*
- KO: *「나리타 도착, 바로 여행 시작. 짐은 호텔에서 만나요.」*

**獲得チャネル:**
1. **Klook / Viator** — 空港送迎カテゴリに「荷物配送」で登録
2. **ホテルコンシェルジュ連携** — チェックイン時に「次回はKAIROXで荷物先送りを」と提案
3. **成田空港内 QR ポスター** — 到着ロビーに掲出（空港の許可取得が必要）
4. **インフルエンサー** — 訪日YouTuber（英語・韓国語・中国語チャンネル）に無料体験提供

### strategy（経営戦略）— 結論

今回の会議で、KAIROX は**単なる空港荷物配送サービス**から**旅行者の旅程全体をカバーする「時間解放プラットフォーム」**へ進化できることが明確になった。

#### コアバリュープロポジション（再定義）

> **「日本に着いた瞬間から、観光が始まる。荷物はKAIROXが届ける。」**
> 円安で旅行予算が豊かな今、旅行者が最も欲しいのはお金ではなく**時間**。KAIROXはその時間を売る。

#### 優先実装ロードマップ

**Phase 1（今週〜2週間）: 価値最大化**

| 機能 | 実装内容 | 工数 |
|------|---------|------|
| ホテル検索強化 | フリーテキスト + 主要50ホテル静的リスト追加 | 1日 |
| 空港間輸送 | 成田↔羽田を DESTINATIONS に追加（¥12,000） | 半日 |
| ドライバー空港情報 | AviationStack API → DriverView に到着便タイムライン | 2日 |
| コピー改訂 | 「到着即観光」訴求に変更 | 半日 |

**Phase 2（1ヶ月）: スケール基盤**

| 機能 | 実装内容 |
|------|---------|
| Google Places API | ホテルオートコンプリート |
| リアルタイムルート | Google Directions API 接続 |

#### リスクへの対応

devils advocateが指摘した「即観光需要の検証」については、**予約時にオプション質問を追加**することで解決する。

```
「荷物を届けた後、どうされますか？」
○ 観光へ直行する
○ ホテルへ向かう
○ その他
```

この1問で行動パターンデータが蓄積され、マーケティング・UX改善に活用できる。
