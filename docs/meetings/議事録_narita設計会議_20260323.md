### planner（計画設計）

現行の5ステップフロー（pickup → destination → luggage → confirm → matching → live）は論理的に整理されている。ただし、ステップ数が多く離脱リスクがある。具体的には、Step 1 で GPS 取得 + ターミナル選択 + フリーテキストと3つの入力モードが並存しており、初見ユーザーへの認知負荷が高い。

**提案:** GPS 取得を最上位の CTA に絞り込み、失敗した場合のみターミナル選択へ誘導する分岐ツリーへの簡素化を推奨する。また、Step 2（配達先選択）は地図カードで視覚的に選ばせると離脱率が下がると予測する。

### research（市場調査）

成田空港の国際線旅行者数は年間約 2,200 万人（2025年実績）。その約 60% が初訪日・英語話者または東アジア圏。既存の手荷物配送サービス（ヤマト・ABC・JAL ABC）は空港カウンターでの対面手続きが必須で、午後便の旅行者は時間的制約により利用を断念するケースが多い。

KAIROXの差別化「ドライバーが来る」モデルは、フィリピン・台湾・韓国で普及している「バイクタクシー型宅配」の利便性を荷物配送に転用したものと評価できる。競合の LuggAgent は成田 T1〜T3 対応だが、配送先が限定的で東京都心 10 エリア対応は KAIROX が先行している。

### engineer（技術設計）

現行アーキテクチャの技術的課題を3点指摘する。

1. **AIルートプランがシミュレーション値** — `calcAiRoute()` 関数の距離・ETA は Math.random() ベース。実際の渋滞・道路状況を反映しない。Google Directions API または Mapbox Route API の接続が Phase 2 の必須課題。

2. **NEXT_PUBLIC_API_URL のクライアント直接呼び出し** — `/narita/page.tsx` から Railway バックエンドへ直接 fetch している。Next.js のサーバーレス API プロキシ（`/api/booking`）経由にすることで、バックエンド URL の非公開化・レート制限・エラーハンドリングの一元化が可能になる。

3. **追跡ページのポーリング（30秒）** — WebSocket または Server-Sent Events（SSE）に切り替えることでリアルタイム性が向上し、サーバー負荷も下がる。現状は許容範囲だが、ユーザー数増加時にボトルネックになる。

### devils advocate（リスク分析・反対意見）

3つの根本的なリスクを提起する。

1. **ドライバー供給の不確実性** — 現在アプリは予約を受け付けるが、実際に配送できるドライバーが何人いるのか不明。予約が入ってもドライバーが見つからない場合、ブランド毀損が致命的になる。「ドライバーを探す」というコンセプト自体が、供給側の準備なしには機能しない。

2. **法的グレーゾーン** — 荷物の一時預かりと陸上輸送は国土交通省の貨物自動車運送事業法の適用範囲。白ナンバー車両での有償輸送は違法。ドライバーが適切な許可を持っているかの確認フローが現仕様に存在しない。

3. **紛失・破損リスク管理** — 「Insured ¥100K」とバッジ表示があるが、実際の保険契約の有無・補償範囲・請求フローが未整備。旅行者の貴重品が紛失した場合の責任の所在が不明確。

### optimizer（改善提案）

devils advocate の指摘を踏まえ、即座に実装できる改善案を提示する。

1. **ドライバー承認フロー** — `/driver/approve/[id]` ページはすでに存在する。Admin画面のドライバー一覧からステータス（承認済み / 待機中 / 稼働中）を管理し、稼働中ドライバーのみがマッチング対象になる仕組みを追加する。

2. **予約前の「ドライバー在庫確認」** — `/api/drivers-active` エンドポイントがすでにある。Step 1 のロード時にアクティブドライバー数を取得し、0件の場合は「現在ドライバーを手配中です。少々お待ちください」表示に差し替えることで、空振り予約を防止できる。

3. **UX改善: プレースホルダー問題** — 現在「Taro Yamada」「+81 / +1…」が入力フィールドのプレースホルダーとして表示されており、入力済みと誤認してボタンが押せないケースが発生した。ラベルを `入力してください` 系に変更するだけで離脱防止になる。

### cto（技術責任者視点）

現時点のスタックは MVP として適切。Next.js + FastAPI + Railway + Vercel の組み合わせはスケールアウトが容易で、コストも最小限に抑えられている。

ただし2点、アーキテクチャ上の判断を今のうちに下しておく必要がある。

**1. 状態管理の設計** — 現在 `/narita/page.tsx` はすべての状態を `useState` で管理する 1,200行超のモノリシックコンポーネント。旅行者数増加・機能追加に伴い、Zustand または React Context への切り出しを Phase 2 で行うべき。

**2. バックエンドのデータベース設計** — 現在 SQLite / PostgreSQL 兼用。Narita エリアの予約データが蓄積されると、相乗りマッチングのクエリパフォーマンスが重要になる。`zone`, `pickup_date`, `status` へのインデックス設計を早期に実施すること。

### marketing（市場戦略）

ターゲットは成田到着の **初訪日インバウンド旅行者**。彼らがアプリを発見するタイミングは以下の3つ。

1. **到着前** — Google「narita luggage delivery」検索 / TripAdvisor / Klook
2. **到着時** — 空港内 QR ポスター / ホテルのコンシェルジュ
3. **到着後** — SNS（インスタグラム・TikTok）での口コミ

現状の Vercel URL（`frontend-psi-seven-15.vercel.app`）では SEO が機能しない。**kairox.jp** または **kairox.com** の独自ドメイン取得が最優先マーケティング施策。

また、`/narita` ページの `<title>` と OGP メタタグが未設定と思われる。これを設定するだけで SNS シェア時のカード表示が改善し、口コミ拡散効果が上がる。

### finance（財務）

現在の料金体系を収益モデルで評価する。

**1件あたり単価:** ¥3,700〜¥7,800（平均 ¥6,000 と仮定）
**ドライバーへの還元率:** 85〜90%（リリースノート記載）
**KAIROX 取り分:** 平均 ¥900 / 件（15%）

**損益分岐点（月次）:**
- 固定費: Railway ¥3,000 + Vercel 無料 + Anthropic API ¥2,000 ≒ ¥5,000/月
- 損益分岐: 6件 / 月（既に軽く超える水準）

**スケール試算:**
| 月次予約件数 | KAIROX 収益（15%） |
|------------|------------------|
| 100件 | ¥90,000 |
| 500件 | ¥450,000 |
| 2,000件 | ¥1,800,000 |

成田空港の日次旅行者数を考慮すると、アクティブドライバー10名体制で月500件は現実的な3ヶ月目標。ただし devils advocate が指摘する法的リスクが顕在化した場合、売上より先に事業停止リスクがある点を強調する。

### strategy（経営戦略）— 結論

各エージェントの議論を踏まえ、以下の戦略的結論をまとめる。

#### 現状評価
KAIROX Narita は **技術的には動作するMVP** として完成している。予約フロー・AI通知・LINE通知・追跡ページまで一気通貫で機能しており、ベータ版として旅行者に使ってもらえるレベルに達している。

#### 最優先課題（今週中）

| 優先度 | 課題 | 担当 |
|--------|------|------|
| P0 | 独自ドメイン取得（kairox.jp） | marketing |
| P0 | 貨物運送許可の確認（または免除要件の確認） | devils advocate |
| P0 | ドライバー1〜3名の確保と稼働テスト | planner |
| P0 | アクティブドライバー表示の実装 | engineer |

#### 2週間アクション

1. **ドライバー確保** → 稼働テスト → 実予約1件成立を目標
2. **独自ドメイン + OGP設定** → SEO・SNS対策
3. **Stripe本番キー** → 実際の決済フロー確認
4. **営業時間制御の本番有効化**

#### 中期戦略（3ヶ月）

成田空港での実績（口コミ・予約件数・追跡データ）を積み上げ、**羽田空港（/haneda）への横展開**を最速で実施する。成田→羽田の乗り継ぎ荷物輸送は単価が高く（¥7,800）、空港間移動という明確なニーズがある。

北海道（千歳）と成田の両拠点が稼働することで、「ドライバーを繋げるプラットフォーム」としてのポジションが確立される。この段階でKKday / Klook への掲載申請と、法人プラン（空港送迎会社・ホテルとのBtoB）展開を並行して進める。
